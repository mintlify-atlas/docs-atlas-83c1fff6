Dokka integrates with Gradle and Maven to generate API reference documentation directly from your source code. This guide walks you through applying the plugin, configuring it for your project structure, and running your first documentation build. If you are upgrading from Dokka Gradle plugin v1, see the [migration guide](/migration) instead.

<Note>
  The Gradle instructions below apply to Dokka Gradle plugin (DGP) v2, which is enabled by default since Dokka 2.1.0.
</Note>

## Gradle

<Steps>
  <Step title="Apply the Dokka plugin">
    Add the Dokka Gradle plugin to the `plugins {}` block in your root `build.gradle.kts` or `build.gradle` file.

    <CodeGroup>

    ```kotlin build.gradle.kts
    plugins {
        id("org.jetbrains.dokka") version "2.2.0"
    }
    ```

    ```groovy build.gradle
    plugins {
        id 'org.jetbrains.dokka' version '2.2.0'
    }
    ```

    </CodeGroup>

    This single declaration is sufficient for single-project builds. Dokka will auto-detect your Kotlin and Java source sets.
  </Step>
  <Step title="Configure multi-project builds (if applicable)">
    When documenting a multi-project build, apply the plugin to every subproject you want to document. The recommended approach is to use a [convention plugin](https://docs.gradle.org/current/userguide/custom_plugins.html#sec:convention_plugins) so that the configuration is shared centrally rather than duplicated across each subproject's build file.

    If you are not using convention plugins, you can also configure each subproject directly in its own `build.gradle.kts` file.

    <Tip>
      See the [Gradle runner documentation](/runners/gradle#multi-project-configuration) for a full walkthrough of multi-project configuration, including how to set up a `buildSrc` convention plugin.
    </Tip>
  </Step>
  <Step title="Generate documentation">
    Run the `dokkaGenerate` task from the root of your project. Always prefix the task with `:` to target the root project specifically.

    ```bash
    ./gradlew :dokkaGenerate
    ```

    For a subproject acting as the aggregating project, include its path:

    ```bash
    ./gradlew :aggregatingProject:dokkaGenerate
    ```

    <Warning>
      Avoid running `./gradlew dokkaGenerate` without the `:` prefix. Without a project path, Gradle attempts to run all `dokkaGenerate` tasks across the entire build, which triggers unnecessary work.
    </Warning>

    By default, Dokka writes the HTML output to `build/dokka/html/`.
  </Step>
  <Step title="View the output">
    Open the generated documentation in your browser:

    ```
    build/dokka/html/index.html
    ```
  </Step>
</Steps>

### Select an output format

By default, Dokka generates HTML documentation. To generate Javadoc output, or both formats simultaneously, apply the appropriate plugin IDs:

```kotlin build.gradle.kts
plugins {
    // Generates HTML documentation
    id("org.jetbrains.dokka") version "2.2.0"

    // Generates Javadoc documentation (Alpha)
    id("org.jetbrains.dokka-javadoc") version "2.2.0"

    // Applying both plugin IDs generates both formats
}
```

Then run the corresponding task:

| Format | Gradle task |
|--------|-------------|
| HTML only | `./gradlew :dokkaGeneratePublicationHtml` |
| Javadoc only | `./gradlew :dokkaGeneratePublicationJavadoc` |
| Both | `./gradlew :dokkaGenerate` |

<Note>
  The Javadoc output format is in Alpha. You may encounter bugs or migration issues when using it with tools that consume Javadoc as input.
</Note>

## Maven

<Steps>
  <Step title="Add the Dokka Maven plugin">
    Add the Dokka Maven plugin to the `plugins` section of your `pom.xml` file:

    ```xml pom.xml
    <build>
        <plugins>
            <plugin>
                <groupId>org.jetbrains.dokka</groupId>
                <artifactId>dokka-maven-plugin</artifactId>
                <version>2.2.0</version>
                <executions>
                    <execution>
                        <phase>pre-site</phase>
                        <goals>
                            <goal>dokka</goal>
                        </goals>
                    </execution>
                </executions>
            </plugin>
        </plugins>
    </build>
    ```

    The plugin is bound to the `pre-site` phase by default, so it runs automatically as part of `mvn site`.
  </Step>
  <Step title="Generate documentation">
    Run the `dokka:dokka` goal directly:

    ```bash
    mvn dokka:dokka
    ```

    Or generate documentation as part of the full site build:

    ```bash
    mvn site
    ```

    By default, the output directory is `target/dokka`.
  </Step>
</Steps>

<Tip>
  For a complete reference of Maven plugin configuration options, see the [Maven runner documentation](/runners/maven).
</Tip>
